What the file is
It's a capability reference document for the Ableton MCP project — essentially a plain-English summary of every tool available and every known limitation, written from the perspective of an AI assistant using these tools.

✅ What it CAN do (50+ tools across 10 categories)
Category	Capabilities
Track & Session	Create MIDI/audio tracks, rename, view session info (tempo, time sig, track count), get detailed track info, delete tracks, duplicate tracks
MIDI Clips	Create clips, add/read/clear notes (pitch, timing, velocity, duration), delete, rename, loop settings, loop points, color, fire/stop, duplicate, quantize, transpose
Playback & Tempo	Start/stop playback, change BPM, fire scenes
Mixing	Volume, pan, mute, solo, arm per track; send levels to return tracks; return track volume/pan/mute/solo; master volume
Instruments & Effects	Load from Ableton's built-in browser, search, browse categories/user library, load samples, load drum kits, delete devices
Device Parameters	Read all parameters for any device, adjust values (filters, envelopes, LFOs, etc.)
Scenes	Create, rename, fire, delete
Clip Automation	Create basic parameter automation within MIDI clips
Browser & Library	Hierarchical browsing, path-based browsing, search, user library, user sample folders
❌ What it CANNOT do
Category	Limitation
Audio Recording/Editing	No real-time recording, no warp/slice/fade editing, no waveform analysis
Arrangement View	Session View only — no arrangement editing or arrangement automation
Advanced Clips	No clip envelopes, no warp markers, no MIDI effects (arpeggiators, scale tools)
Third-party Plugins	Cannot browse or load VST/AU plugins, cannot create/modify presets
Advanced Mixing	No pre/post fader routing, no custom effect chains/groups, no freeze/flatten
Advanced MIDI	No humanization, no groove templates, no MPE/polyphonic expression
Project Management	Cannot save/load/export projects, no multi-set management, no audio/MIDI export
Max for Live	No interaction with M4L devices or custom instruments
In short

Plan: Max for Live Bridge Integration
Goal
Add an optional Max for Live (M4L) bridge device that runs alongside the existing Remote Script to unlock three capabilities:

M4L as a communication bridge — a .amxd device that listens for commands via UDP, complementing the existing TCP Remote Script
Hidden Operator parameters — access non-automatable parameters (algorithm, oscillator waveforms, FM matrix) via the Live Object Model (LOM)
VST/AU loading — document that this is a hard API limitation (no fix exists), and implement a convenience workflow using Instrument Rack wrappers
Research Conclusions
Capability	Feasible?	Method
M4L bridge device	Yes	udpreceive + js object with LiveAPI
Hidden Operator params	Yes	M4L LiveAPI accesses full LOM parameter tree
VST/AU programmatic loading	No	Not possible through any Ableton API (Remote Script, M4L, or LOM)
Architecture
Claude/Cursor (MCP Client)
    │
    │ MCP Protocol (stdio)
    ▼
MCP Server (server.py)
    │
    ├── TCP :9877 ──────────► Remote Script (existing, unchanged)
    │                              │
    │                              ▼
    │                         Ableton Live (public API)
    │
    └── UDP :9878 / :9879 ──► M4L Device (NEW .amxd)
                                   │
                                   ▼
                              Ableton Live (deep LOM, hidden params)
Backward compatible: If M4L device is not loaded, all existing tools work unchanged. New M4L tools return a clear error.
Separate ports: Remote Script keeps TCP :9877. M4L uses UDP :9878 (commands) and :9879 (responses).
No Remote Script changes needed.

To test the M4L bridge, here's what you need to do:

Step 1: Build the .amxd Device in Ableton
Open Ableton Live (needs Suite or Standard + Max for Live)
Create a new MIDI track
Drag a "Max MIDI Effect" from the browser onto that track
Click the wrench icon to open the Max editor
Build this patch — press N to create each object, connect them top to bottom:
[udpreceive 9878]
       |
[fromsymbol]
       |
[js m4l_bridge.js]
       |
[tosymbol]
       |
[udpsend localhost 9879]

Copy m4l_bridge.js from M4L_Device/m4l_bridge.js into the Max project's folder (Max should show you where — check the sidebar file browser in the Max editor)
Save the device as AbletonMCP_Bridge.amxd
Lock the patch (Ctrl+E) and close the editor
Step 2: Make Sure the Remote Script Is Running
Your existing AbletonMCP Remote Script should already be loaded (the one on port 9877).

Step 3: Start the MCP Server
cd C:\Users\akioo\Downloads\ableton-mcp-beta
python -m MCP_Server.server

Step 4: Test the Tools
From Claude/Cursor (or any MCP client), try these in order:

m4l_status() — should return "M4L bridge connected (v1.0.0)"
get_session_info() — verify existing tools still work (regression check)
discover_device_params(track_index=0, device_index=0) — load any instrument on track 0 first, then run this to see ALL parameters including hidden ones
Compare: Run get_device_parameters(track_index=0, device_index=0) (existing tool) vs get_device_hidden_parameters(track_index=0, device_index=0) (new tool) — the new one should show more parameters
set_device_hidden_parameter(track_index=0, device_index=0, parameter_index=X, value=Y) — pick an index from the discovery results and change it, verify it moves in Ableton's GUI
list_instrument_rack_presets() — shows any saved racks in your user library
Quick Smoke Test Without M4L
You can also verify that everything degrades gracefully without the M4L device loaded:

m4l_status() → should return a clear "not connected" message
All existing tools → should work exactly as before
discover_device_params() → should return "M4L bridge not available" error
The key thing to validate is that Operator (or Wavetable, Analog, etc.) shows more parameters through discover_device_params than through the standard get_device_parameters.
